# 📏 工匠守则：认知契约 (The Code of Craftsmanship)

> **"Programs must be written for people to read, and only incidentally for machines to execute."**  
> —— Harold Abelson, *SICP*

---

## 🧠 导读：代码规范的本质

为什么我们需要代码规范？

不是为了满足强迫症，不是为了让代码看起来像阅兵方阵。
代码规范的本质是**降低认知负荷 (Cognitive Load)**。

当你阅读代码时，你的大脑在进行复杂的解码工作。如果代码风格混乱，大脑就需要消耗额外的能量去解析格式，剩下的能量就不够理解业务逻辑了。
**一致性 (Consistency)** 创造了**预测性 (Predictability)**。当代码符合你的预测时，它就变得"透明"了，你可以直接看到逻辑本身。

这份文档不是束缚你的枷锁，而是团队成员之间签订的一份**认知契约**。

## 🏷️ 第一章：命名 —— 思维的压缩算法

计算机科学中最难的两件事是：缓存失效和**命名**。—— Phil Karlton

命名不仅仅是给变量贴标签，它是**设计的微缩模型**。一个好的名字是一个高效的压缩算法，它能让阅读者在不看实现细节的情况下，瞬间解压出变量的类型、作用域和意图。

### 1.1 骆驼的驼峰 (CamelCase) 与生态的方言

*   **规则**：
    *   **变量、函数、方法**：使用 `camelCase` (小驼峰)。
        *   e.g., `userProfile`, `fetchData`, `isValid`
    *   **类、接口、类型、React 组件**：使用 `PascalCase` (大驼峰)。
        *   e.g., `UserService`, `UserProfile`, `SubmitButton`
    *   **常量（真正的不可变值）**：使用 `UPPER_SNAKE_CASE` (全大写下划线)。
        *   e.g., `MAX_RETRY_COUNT`, `API_BASE_URL`
    *   **文件名**：
        *   组件文件：`UserProfile.tsx` (PascalCase)
        *   工具/逻辑文件：`dateUtils.ts` (camelCase) 或 `date-utils.ts` (kebab-case) —— **关键是项目内保持一致**。

*   **哲学解读**：
    这不是审美问题，这是**方言 (Dialect)**。在 JavaScript 的国度里，遵守当地的方言能让你的代码"隐形"。
    *   *Bad*: `user_name` (这是 Python 或 SQL 的口音，会引起大脑的微小停顿)
    *   *Good*: `userName` (大脑瞬间识别：这是一个 JS 变量)
    *   *Bad*: `const timeout = 5000;` (这看起来像个变量，可能会变)
    *   *Good*: `const DEFAULT_TIMEOUT = 5000;` (大写咆哮着告诉你：我是配置，我不变！)

### 1.2 布尔值的谓词性 (The Predicate Nature)

*   **规则**：布尔值变量必须回答 "Yes/No" 的问题。使用 `is`, `has`, `can`, `should`, `did` 等前缀。
    *   `isLoading`: 是否正在加载？
    *   `hasPermission`: 是否拥有权限？
    *   `canEdit`: 是否可以编辑？
    *   `shouldUpdate`: 是否应该更新？

*   **哲学解读**：
    布尔值代表状态的**断言**。
    *   *Bad*: `const open = true;` (是"打开"这个动作？还是"开着的"状态？歧义即 Bug 之源。)
    *   *Good*: `const isOpen = true;` (毫无歧义的状态描述。)
    *   *Bad*: `const status = true;` (什么状态？成功？激活？在线？)

### 1.3 诚实原则 (The Principle of Honesty)

*   **规则**：函数名必须完全覆盖其副作用。
*   **哲学解读**：
    如果一个叫 `getUser()` 的函数顺便修改了数据库，它就在撒谎。撒谎的代码是不可维护的。
    *   *Bad*: `getUser()` (内部却更新了 `lastLoginTime`)
    *   *Good*: `getUserAndUpdateLoginTime()` (虽然长，但是诚实)
    *   *Better*: 拆分成两个函数 `getUser()` 和 `updateLoginTime()` (单一职责)

### 1.4 具体的艺术

*   **规则**：拒绝泛泛而谈的名字。
*   **哲学解读**：
    `data`, `info`, `item`, `value`, `handle` 这些词是**语义黑洞**。它们没有提供任何信息。
    *   *Bad*: `handleData(data)`
    *   *Good*: `parseUserProfile(jsonString)`
    *   *Hint*: 如果你发现很难给一个变量命名，通常意味着**你的设计有问题**（这个变量承载了太多含义）。

## 🎨 第二章：格式化 —— 视觉的韵律

代码的形状 (Shape of Code) 决定了阅读的快慢。

### 2.1 缩进与留白

*   **规则**：
    *   使用 **2 个空格** 缩进（JS 社区标准）。
    *   在逻辑块之间留出**空行**。
*   **哲学解读**：
    代码是二维的文本。缩进表达了**逻辑的层级**，空行表达了**逻辑的段落**。
    *   不要吝啬空行。就像写文章分段一样，把相关的代码聚在一起，不相关的隔开。
    *   *Bad*: 一坨 50 行没有任何空行的代码块。这是"视觉窒息"。

### 2.2 括号与分号

*   **规则**：
    *   **始终**使用分号（避免 ASI 机制带来的"灵异事件"）。
    *   `if`, `for`, `while` 语句块**始终**使用大括号 `{}`，即使只有一行。
*   **哲学解读**：
    **防御性编码**。
    *   省略大括号看起来很酷很极简，直到有一天你注释掉了那一行，或者想在下面加一行 log，然后逻辑全崩了。
    *   显式优于隐式。

### 2.3 引号与逗号

*   **规则**：
    *   **引号**：优先使用单引号 `'`，需要插值时使用反引号 `` ` ``。保持一致。
    *   **尾随逗号 (Trailing Comma)**：在多行对象/数组定义中，最后一行**加上逗号**。
*   **哲学解读**：
    *   **尾随逗号**：这看起来很多余，但它是为了 **Git Diff** 服务的。当你添加新行时，不需要修改上一行（添加逗号），从而保持 Diff 的纯净。

### 2.4 行宽

*   **规则**：单行不超过 80-100 字符。
*   **哲学解读**：
    人类的眼睛不擅长长距离扫描。报纸分栏是有原因的。
    *   如果一行代码太长，说明它可能干了太多的事。尝试提取变量，或者换行。

## 📝 第三章：注释 —— 意图的传递

注释是代码的**元数据 (Metadata)**。

### 3.1 Why > How

*   **规则**：注释应该解释**为什么**这么做，而不是**代码在做什么**。
*   **哲学解读**：
    代码本身已经展示了"它在做什么"（如果代码展示不清楚，那就重构代码）。
    注释应该填补"代码无法表达的上下文"。
    *   *Bad*: `// i 加 1` (废话，我看得懂 `i++`)
    *   *Good*: `// 使用位运算代替除法，性能提升 20% (参考 issue #123)` (解释了非直观实现的理由)
    *   *Good*: `// FIXME: 这里的算法复杂度是 O(n^2)，数据量大时需要优化` (承认缺陷，标记技术债)

### 3.2 自文档化的谎言

*   **观点**：有人说"好的代码不需要注释"。这是一个半真半假的谎言。
*   **真相**：
    好的代码不需要注释来解释"它在干什么"。
    但即使是最好的代码，也无法解释**业务规则的怪癖**、**历史遗留的坑**、或者**复杂的数学公式**。
    *   不要因为追求"自文档化"而拒绝写必要的注释。

### 3.3 JSDoc / TSDoc

*   **规则**：公共接口（Public API）必须有文档注释。
*   **哲学解读**：
    你的代码会被别人（或者三个月后的你）调用。JSDoc 是**使用说明书**。
    *   描述参数的约束（e.g., "必须是正整数"）。
    *   描述返回值的承诺。
    *   描述可能抛出的异常。

```javascript
/**
 * 计算订单总价
 * @param {number[]} prices - 商品价格列表，必须非空
 * @returns {number} 总价
 * @throws {Error} 如果 prices 为空
 */
function calculateTotal(prices) { ... }
```

### 3.4 特殊标记

*   **规则**：使用标准化的标记来沟通代码状态。
    *   `TODO`: 待办事项（功能未完成）。
    *   `FIXME`: 已知缺陷（代码能跑，但有 Bug 或性能问题）。
    *   `HACK`: 权宜之计（为了绕过某个 Bug 而写的丑陋代码，解释原因）。
    *   `DEPRECATED`: 已废弃（不要再使用）。

## 🏗️ 第四章：结构 —— 分形的秩序

### 4.1 文件的解剖学

*   **规则**：保持文件结构的统一性。
    1.  **Imports**: 第三方库 -> 本地库 -> 样式 (逻辑清晰的依赖链)
    2.  **Types / Interfaces**: 定义数据形状
    3.  **Constants**: 定义魔法数字
    4.  **Main Component / Function**: 文件的核心
    5.  **Helper Functions**: 辅助逻辑
*   **哲学解读**：
    当你打开一个新文件时，你应该能立刻知道去哪里找东西。这种**空间记忆**能极大地提升导航速度。

### 4.2 模块化原则

*   **规则**：一个文件只做一个主要任务（Single Responsibility Principle）。
*   **哲学解读**：
    如果一个文件超过 300 行，它通常在尖叫："把我拆分！"
    小文件易于阅读，易于测试，易于复用。

## 🛡️ 第五章：工具链 —— 机器警察

不要靠人眼去检查规范。人眼是不可靠的，而且把时间浪费在争论"是否应该加空格"上是愚蠢的。

### 5.1 Prettier & ESLint

*   **策略**：
    *   **Prettier**: 负责格式 (Formatting)。它是独裁者，不容置疑。保存即格式化。忘掉缩进，忘掉空格，把排版交给机器。
    *   **ESLint**: 负责质量 (Quality)。它检查潜在的 Bug，未使用的变量，危险的写法。
*   **哲学解读**：
    **自动化**是规范落地的唯一保障。如果规范只存在于文档里，它就是废纸。只有配置在 CI/CD 流水线里的规范才是真正的规范。

### 5.2 配置示例

**ESLint (.eslintrc.js)**:
```javascript
module.exports = {
  extends: ['eslint:recommended'],
  rules: {
    'no-unused-vars': 'warn', // 警告未使用的变量，防止代码腐烂
    'eqeqeq': 'error',        // 强制使用 ===，避免类型转换陷阱
    'no-console': 'warn',     // 生产环境不应有 console.log
  }
};
```

**Prettier (.prettierrc)**:
```json
{
  "semi": true,
  "singleQuote": true,
  "trailingComma": "es5",
  "printWidth": 80
}
```

## 🏁 结语

代码规范不是为了限制你的创造力。
恰恰相反，通过将琐碎的格式和命名决策**标准化**、**自动化**，你的大脑被解放出来，可以去思考真正重要的问题：
**架构、算法、业务价值。**

遵守契约，成为工匠。
